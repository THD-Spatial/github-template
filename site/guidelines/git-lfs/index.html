    Git LFS (Large File Storage)¶

    Git LFS is essential for managing large files in your repository, particularly data files common in spatial research projects.

    Why Git LFS?¶

    Standard Git is designed for text files and struggles with large binary files:

    ❌ Without Git LFS: - Repository becomes bloated - Clone/fetch operations are slow - Every version of large files stored in history - GitHub has file size limits (100 MB per file)

    ✅ With Git LFS: - Large files stored separately - Fast clone/fetch operations - Only download files when needed - Efficient storage of file versions

    Important

    Git LFS is required for THD-Spatial repositories that contain large data files.

    When to Use Git LFS¶

    Use Git LFS for:

    • Data files: NetCDF, HDF5, GeoTIFF, Zarr
    • Media files: Images, videos, audio
    • Binary files: Compiled code, archives
    • Large documents: Large PDFs, presentations
    • Machine learning models: Model weights, checkpoints

    Typical spatial data file formats:

    *.nc          # NetCDF files
*.tif, *.tiff # GeoTIFF files
*.hdf, *.h5   # HDF5 files
*.zarr        # Zarr stores
*.shp         # Shapefiles
*.gdb         # Geodatabases
*.las, *.laz  # LiDAR data

    Installation¶

    Linux (Ubuntu/Debian)¶

    sudo apt-get install git-lfs
git lfs install

    macOS¶

    brew install git-lfs
git lfs install

    Windows¶

    1. Download installer from git-lfs.github.com
    2. Run the installer
    3. Open Git Bash and run: git lfs install

    Verify Installation¶

    git lfs version
# Output: git-lfs/3.4.0 (GitHub; linux amd64; go 1.21.0)

    Basic Usage¶

    Adding LFS to Existing Repository¶

    # Navigate to your repository
cd your-repo

# Install Git LFS for this repository
git lfs install

# Track file types
git lfs track "*.nc"
git lfs track "*.tif"
git lfs track "*.h5"

# Add .gitattributes (created by lfs track)
git add .gitattributes
git commit -m "Configure Git LFS"

    Track Specific Files¶

    # Track by extension
git lfs track "*.nc"

# Track specific file
git lfs track "data/large_dataset.tif"

# Track all files in directory
git lfs track "data/**"

# Track by pattern
git lfs track "**/satellite_*.tif"

    View Tracked Patterns¶

    # List tracked patterns
git lfs track

# Check if file is tracked
git lfs ls-files

    Adding Files¶

    Once tracking is configured, add files normally:

    # Add large file (will be handled by LFS)
git add data/large_file.nc
git commit -m "Add large dataset"
git push origin main

    Configuration File¶

    Git LFS uses .gitattributes to track patterns:

    # NetCDF files
*.nc filter=lfs diff=lfs merge=lfs -text
*.nc4 filter=lfs diff=lfs merge=lfs -text

# GeoTIFF files
*.tif filter=lfs diff=lfs merge=lfs -text
*.tiff filter=lfs diff=lfs merge=lfs -text

# HDF5 files
*.hdf filter=lfs diff=lfs merge=lfs -text
*.hdf5 filter=lfs diff=lfs merge=lfs -text
*.h5 filter=lfs diff=lfs merge=lfs -text

# Zarr stores
*.zarr/** filter=lfs diff=lfs merge=lfs -text

# Shapefiles
*.shp filter=lfs diff=lfs merge=lfs -text
*.shx filter=lfs diff=lfs merge=lfs -text
*.dbf filter=lfs diff=lfs merge=lfs -text

# LiDAR data
*.las filter=lfs diff=lfs merge=lfs -text
*.laz filter=lfs diff=lfs merge=lfs -text

# Images
*.png filter=lfs diff=lfs merge=lfs -text
*.jpg filter=lfs diff=lfs merge=lfs -text
*.jpeg filter=lfs diff=lfs merge=lfs -text

# Archives
*.zip filter=lfs diff=lfs merge=lfs -text
*.tar.gz filter=lfs diff=lfs merge=lfs -text

# Model files
*.h5 filter=lfs diff=lfs merge=lfs -text
*.pkl filter=lfs diff=lfs merge=lfs -text
*.model filter=lfs diff=lfs merge=lfs -text

    Common Workflows¶

    Cloning LFS Repository¶

    # Clone repository (LFS files downloaded automatically)
git clone https://github.com/THD-Spatial/your-repo.git

# Or clone without downloading LFS files initially
GIT_LFS_SKIP_SMUDGE=1 git clone https://github.com/THD-Spatial/your-repo.git

# Then download specific LFS files
git lfs pull --include="data/*.nc"

    Downloading LFS Files¶

    # Download all LFS files
git lfs pull

# Download specific files
git lfs pull --include="data/specific_file.nc"

# Download files matching pattern
git lfs pull --include="data/*.tif"

# Exclude certain files
git lfs pull --exclude="data/temp/*"

    Checking LFS Status¶

    # List LFS files
git lfs ls-files

# Show LFS environment
git lfs env

# Show storage usage
git lfs status

    Migrating Existing Files¶

    If you have large files already committed:

    # Migrate existing files to LFS
git lfs migrate import --include="*.nc"

# Migrate and rewrite history
git lfs migrate import --include="*.nc" --everything

# Force push after migration
git push origin --force --all

    Warning

    Rewriting history with --everything changes commit hashes. Coordinate with collaborators.

    GitHub Integration¶

    Storage Limits¶

    GitHub provides: - 1 GB free LFS storage per account - 1 GB/month free bandwidth - Additional data packs available

    Checking Usage¶

    View LFS usage in GitHub repository settings under "Billing".

    Release Assets¶

    Critical

    Files tracked by Git LFS are not included in release source archives.

    To include LFS files in releases:

    # Create release and upload LFS files separately
gh release create v1.0.0 \
  --title "Release 1.0.0" \
  --notes "Release notes" \
  data/dataset.nc data/model.h5

    Or use GitHub Actions:

    name: Release
on:
  release:
    types: [created]

jobs:
  upload-assets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          lfs: true

      - name: Upload LFS files to release
        uses: softprops/action-gh-release@v1
        with:
          files: |
            data/*.nc
            data/*.tif

    Best Practices¶

    DO:¶

    ✅ Track all files over 10 MB ✅ Commit .gitattributes file ✅ Use consistent patterns ✅ Document which files are LFS-tracked ✅ Verify LFS tracking before committing: git lfs ls-files ✅ Include LFS setup in installation instructions

    DON'T:¶

    ❌ Track small files (<1 MB) with LFS (unnecessary overhead) ❌ Forget to run git lfs install on new machines ❌ Mix LFS and non-LFS tracking for same file type ❌ Assume LFS files will be in release archives ❌ Track frequently-changing files (if possible)

    Troubleshooting¶

    Files Not Tracked¶

    # Check if pattern is tracked
git lfs track

# Manually mark file as LFS
git lfs track "path/to/file.nc"
git add .gitattributes path/to/file.nc
git commit --amend

    Bandwidth Exceeded¶

    # Clone without LFS files
GIT_LFS_SKIP_SMUDGE=1 git clone <repo>

# Download only needed files
git lfs pull --include="data/small_dataset.nc"

    Fixing Already-Committed Large Files¶

    # Track with LFS
git lfs track "*.nc"

# Migrate existing files
git lfs migrate import --include="*.nc" --everything

# Force push (warn collaborators first!)
git push origin --force --all

    LFS Pointer Instead of File¶

    If you see pointer file content:

    version https://git-lfs.github.com/spec/v1
oid sha256:abc123...
size 1234567

    Run: git lfs pull

    Documentation in README¶

    Include LFS requirements in your README:

    ## Installation

### Prerequisites

This repository uses Git LFS for large data files. Install Git LFS before cloning:

\`\`\`bash
# Ubuntu/Debian
sudo apt-get install git-lfs

# macOS
brew install git-lfs

# Initialize Git LFS
git lfs install
\`\`\`

### Clone Repository

\`\`\`bash
git clone https://github.com/THD-Spatial/your-repo.git
cd your-repo

# Verify LFS files downloaded
git lfs ls-files
\`\`\`

## Data Files

The following data files are managed with Git LFS:
- `data/*.nc` - Climate datasets
- `data/*.tif` - Satellite imagery
- `models/*.h5` - Trained model weights

    Checklist¶

    Before finalizing your Git LFS setup:

    • [ ] Git LFS installed on local machine
    • [ ] git lfs install run in repository
    • [ ] Appropriate file patterns tracked
    • [ ] .gitattributes file committed
    • [ ] Large files added and pushed successfully
    • [ ] LFS files verified: git lfs ls-files
    • [ ] LFS requirements documented in README
    • [ ] Team members informed about LFS requirements
    • [ ] Release strategy for LFS files planned (if applicable)
    • [ ] GitHub LFS storage/bandwidth sufficient
