    Contributing Guidelines¶

    Clear contributing guidelines help potential contributors understand how to participate in your project effectively.

    Why Contributing Guidelines Matter¶

    Contributing guidelines:

    • Lower barriers - Make it easy for newcomers to contribute
    • Set expectations - Define what's acceptable and how to participate
    • Maintain quality - Establish standards for contributions
    • Save time - Reduce back-and-forth by providing clear instructions

    What to Include¶

    A good CONTRIBUTING.md file should cover:

    1. ✅ How to report bugs
    2. ✅ How to suggest enhancements
    3. ✅ How to submit code changes
    4. ✅ Coding standards
    5. ✅ Commit message conventions
    6. ✅ Testing requirements
    7. ✅ Review process

    Template Structure¶

    Here's a comprehensive template for CONTRIBUTING.md:

    # Contributing to [Project Name]

Thank you for your interest in contributing! We welcome contributions from the community.

## Code of Conduct

This project adheres to a [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## How Can I Contribute?

### Reporting Bugs

Before creating bug reports, please check existing issues to avoid duplicates.

**How to submit a bug report:**

1. Use the issue tracker
2. Use a clear and descriptive title
3. Describe the exact steps to reproduce
4. Provide specific examples
5. Describe the observed behavior vs expected behavior
6. Include screenshots if applicable
7. Include your environment details (OS, Python version, etc.)

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues.

**How to submit an enhancement:**

1. Use the issue tracker
2. Use a clear and descriptive title
3. Provide a detailed description of the proposed functionality
4. Explain why this enhancement would be useful
5. List any similar features in other projects

### Pull Requests

1. **Fork** the repository
2. **Create a branch** for your changes: `git checkout -b feature/my-feature`
3. **Make your changes** following the coding standards below
4. **Write tests** for your changes
5. **Run tests** to ensure everything passes
6. **Commit** with descriptive messages
7. **Push** to your fork: `git push origin feature/my-feature`
8. **Open a Pull Request** with a clear description

## Development Setup

\`\`\`bash
# Clone your fork
git clone https://github.com/your-username/project-name.git
cd project-name

# Create virtual environment
python -m venv venv
source venv/bin/activate  # On Windows: venv\\Scripts\\activate

# Install dependencies
pip install -r requirements.txt
pip install -r requirements-dev.txt

# Install project in editable mode
pip install -e .
\`\`\`

## Coding Standards

### Python Style Guide

- Follow [PEP 8](https://pep8.org/)
- Use 4 spaces for indentation (no tabs)
- Maximum line length: 88 characters (Black formatter default)
- Use meaningful variable and function names

### Code Formatting

We use automated formatters:

\`\`\`bash
# Format code with Black
black .

# Sort imports with isort
isort .

# Check types with mypy (if applicable)
mypy src/
\`\`\`

### Documentation

- Add docstrings to all public functions and classes
- Use Google-style or NumPy-style docstrings
- Update README.md if adding new features
- Add comments for complex logic

Example docstring:

\`\`\`python
def process_data(data: pd.DataFrame, threshold: float = 0.5) -> pd.DataFrame:
    """
    Process input data by applying threshold filtering.

    Args:
        data: Input DataFrame containing raw data
        threshold: Minimum value to retain (default: 0.5)

    Returns:
        Filtered DataFrame with values above threshold

    Raises:
        ValueError: If threshold is negative
    """
    pass
\`\`\`

## Commit Message Guidelines

Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

\`\`\`
<type>(<scope>): <subject>

<body>

<footer>
\`\`\`

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Formatting changes (no code change)
- `refactor`: Code refactoring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples:**

\`\`\`
feat(parser): add support for GeoTIFF files

fix(processing): handle missing values correctly

docs: update installation instructions

test(utils): add tests for coordinate transformation
\`\`\`

## Testing

All code changes should include tests.

### Running Tests

\`\`\`bash
# Run all tests
pytest

# Run with coverage
pytest --cov=src --cov-report=html

# Run specific test file
pytest tests/test_feature.py
\`\`\`

### Writing Tests

\`\`\`python
import pytest
from myproject import my_function

def test_my_function_basic():
    """Test basic functionality."""
    result = my_function(input_data)
    assert result == expected_output

def test_my_function_edge_case():
    """Test edge case handling."""
    with pytest.raises(ValueError):
        my_function(invalid_input)
\`\`\`

## Pull Request Review Process

1. **Automated checks** must pass (tests, linting, type checking)
2. **Code review** by at least one maintainer
3. **Discussion** if changes are needed
4. **Merge** once approved

### What We Look For

- ✅ Code follows style guidelines
- ✅ Tests are included and passing
- ✅ Documentation is updated
- ✅ Commit messages are clear
- ✅ Changes are focused and well-scoped
- ✅ No breaking changes (or clearly documented)

## Getting Help

- 💬 Open an issue for questions
- 📧 Contact maintainers: [email@example.com]
- 📖 Check existing documentation
- 🔍 Search closed issues for similar questions

## Recognition

Contributors will be:
- Listed in CONTRIBUTORS.md
- Credited in release notes
- Shown appreciation for their efforts

Thank you for contributing!

- Follow tidyverse style guide
- Use styler package

- Follow Airbnb style guide
- Use ESLint

    Contributor License Agreement (CLA)¶

    For larger projects, consider a CLA:

    ## Contributor License Agreement

By submitting a pull request, you agree that:
- You have the right to submit the contribution
- You grant the project a perpetual, worldwide license to use your contribution
- Your contribution is provided under the project's license

    Branch Strategy¶

    Define your branching model:

    ## Branch Strategy

- `main`: Production-ready code
- `develop`: Integration branch
- `feature/*`: New features
- `bugfix/*`: Bug fixes
- `hotfix/*`: Urgent production fixes
