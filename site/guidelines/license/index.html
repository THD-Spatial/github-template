    License Guidelines¶

    Why Licensing Matters¶

    A license is required for all public repositories in the THD-Spatial organization. Without a license, your code is under exclusive copyright by default, meaning others cannot legally use, copy, modify, or distribute your work.

    Critical Requirement

    You cannot make your repository public under THD-Spatial without including a LICENSE file.

    Choosing a License¶

    Different licenses suit different needs. Here's a guide to help you choose:

    Permissive Licenses¶

    Allow maximum freedom for users with minimal restrictions.

    MIT License¶

    • Best for: Most projects, especially libraries
    • Allows: Commercial use, modification, distribution, private use
    • Requires: License and copyright notice
    • Limitations: No liability or warranty
    ✅ Simple and permissive
✅ Most commonly used license
✅ Easy to understand
✅ Compatible with most projects

    Apache License 2.0¶

    • Best for: Projects concerned with patent rights
    • Allows: Same as MIT, plus explicit patent grant
    • Requires: License notice, state changes, copyright notice
    • Limitations: No liability, warranty, or trademark use
    ✅ Includes patent protection
✅ Good for projects with contributors
✅ Widely used in industry
⚠️ More complex than MIT

    BSD 3-Clause License¶

    • Best for: Academic and research projects
    • Allows: Commercial use, modification, distribution
    • Requires: License notice, copyright notice
    • Limitations: Cannot use contributors' names for endorsement
    ✅ Simple and well-established
✅ Common in academia
✅ Similar to MIT

    Copyleft Licenses¶

    Require derivative works to use the same license.

    GNU GPL v3¶

    • Best for: Projects you want to keep open source
    • Allows: Commercial use, modification, distribution
    • Requires: Disclose source, same license, state changes
    • Limitations: Strong copyleft - derivatives must be GPL
    ⚠️ Strong copyleft requirement
⚠️ May limit adoption in commercial projects
✅ Ensures derivatives remain open source
✅ Good for preventing proprietary forks

    GNU LGPL v3¶

    • Best for: Libraries that might be used in proprietary software
    • Allows: Linking from proprietary code
    • Requires: Disclose source of library, same license for library
    • Limitations: Modifications to library must be LGPL
    ✅ More permissive than GPL
✅ Allows use in proprietary software
⚠️ More complex licensing requirements

    Creative Commons¶

    For non-software content (documentation, datasets, media).

    CC BY 4.0¶

    • Best for: Documentation, datasets, educational materials
    • Allows: Sharing and adaptation with attribution
    • Requires: Attribution, indicate if changes made
    ✅ Good for data and documentation
✅ Requires credit to original author
⚠️ Not recommended for software code

    Recommended Licenses by Project Type¶

    Project Type Recommended License Why
    Python/R Library MIT or Apache 2.0 Maximum adoption, simple
    Data Processing Tool MIT Easy for users to integrate
    Research Dataset CC BY 4.0 Appropriate for data
    Full Application MIT or GPL-3.0 MIT for permissive, GPL to ensure openness
    Documentation CC BY 4.0 Standard for documentation

    How to Add a License¶

    Step 1: Choose Your License¶

    Visit choosealicense.com for an interactive guide, or use the table above.

    Step 2: Create LICENSE File¶

    1. Create a file named LICENSE (no extension) in your repository root
    2. Copy the full license text
    3. Update placeholders:
    4. [year] → Current year (e.g., 2026)
    5. [fullname] → Your name or "THD-Spatial Group"
    6. [email] → Contact email (if required)

    Step 3: Reference in README¶

    Add a license section to your README.md:

    ## License

This project is licensed under the MIT License - see the [LICENSE](LICENSE) file for details.

    Step 4: Add License Headers (Optional)¶

    For larger projects, add license headers to source files:

    # Copyright (c) 2026 THD-Spatial Group
# Licensed under the MIT License - see LICENSE file for details

    Common Questions¶

    Can I change the license later?¶

    You can change to a more permissive license, but making it more restrictive is problematic since others may have already used your code under the permissive license.

    Can I use multiple licenses?¶

    Yes, you can dual-license or use different licenses for different parts (e.g., MIT for code, CC BY for data).

    What if I use code from other projects?¶

    You must comply with their licenses. MIT and Apache are usually compatible, but GPL code requires your project to be GPL.

    Do I need permission from all contributors?¶

    If others have contributed code, they hold copyright too. Get agreement before changing licenses, or use a Contributor License Agreement (CLA).

    License Compatibility¶

    When using dependencies, ensure licenses are compatible:

    MIT ✅ Compatible with: Most licenses
Apache 2.0 ✅ Compatible with: GPL-3.0+, MIT, BSD
GPL-3.0 ⚠️ Requires: Derivative work to be GPL-3.0
BSD ✅ Compatible with: Most licenses

    Checklist¶

    Before proceeding, ensure:

    • [ ] License chosen based on project needs
    • [ ] LICENSE file created in repository root
    • [ ] Year and copyright holder updated
    • [ ] License mentioned in README.md
    • [ ] All dependencies' licenses checked for compatibility
    • [ ] All contributors agree (if multiple contributors)
