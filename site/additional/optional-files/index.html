    Optional Files¶

    While not required, these additional files can significantly improve your repository's professionalism, usability, and community engagement.

    Overview¶

    Optional files help with:

    • Project management - Track changes and ownership
    • User experience - Help users get support and report issues effectively
    • Security - Handle vulnerability reports responsibly
    • Automation - Streamline workflows with GitHub Actions

    CHANGELOG.md¶

    Purpose¶

    Document all notable changes to your project in a human-readable format.

    Why Include It?¶

    • ✅ Users can quickly see what changed between versions
    • ✅ Helps maintain clear release notes
    • ✅ Shows project is actively maintained
    • ✅ Aids in debugging version-specific issues

    Format¶

    Follow Keep a Changelog format:

    # Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- New feature X

### Changed
- Improved performance of Y

## [1.1.0] - 2026-02-15

### Added
- Feature A for doing X
- Feature B for handling Y

### Fixed
- Bug in data processing
- Memory leak in Z module

### Changed
- Updated dependencies

## [1.0.0] - 2026-01-01

### Added
- Initial release
- Core functionality
- Basic documentation

[Unreleased]: https://github.com/user/repo/compare/v1.1.0...HEAD
[1.1.0]: https://github.com/user/repo/compare/v1.0.0...v1.1.0
[1.0.0]: https://github.com/user/repo/releases/tag/v1.0.0

    Change Categories¶

    • Added: New features
    • Changed: Changes in existing functionality
    • Deprecated: Soon-to-be removed features
    • Removed: Removed features
    • Fixed: Bug fixes
    • Security: Security vulnerability fixes

    Automation¶

    Generate changelog from commits:

    # Using github-changelog-generator
gem install github_changelog_generator
github_changelog_generator

# Using git-chglog
brew install git-chglog
git-chglog

    CODEOWNERS¶

    Purpose¶

    Define individuals or teams responsible for code in your repository.

    Why Include It?¶

    • ✅ Automatic reviewer assignment on PRs
    • ✅ Clear ownership of different components
    • ✅ Helps distribute review workload
    • ✅ Ensures domain experts review relevant changes

    Location¶

    Place in: - .github/CODEOWNERS - CODEOWNERS (root) - docs/CODEOWNERS

    Format¶

    # Default owner for everything
* @THD-Spatial/maintainers

# Documentation
/docs/ @THD-Spatial/docs-team
*.md @THD-Spatial/docs-team

# Python code
*.py @THD-Spatial/python-developers
/src/core/ @THD-Spatial/core-team

# Specific modules
/src/data/ @username1 @username2
/src/models/ @username3

# Configuration
/.github/ @THD-Spatial/admin
*.yml @THD-Spatial/admin
*.yaml @THD-Spatial/admin

# Data files
/data/ @THD-Spatial/data-team

    Patterns¶

    # Match all files
*

# Match specific extension
*.js

# Match directory
/docs/

# Match files in all subdirectories
docs/**/*.md

# Match nested path
/src/components/*/tests/

    Issue Templates¶

    Purpose¶

    Provide structured formats for bug reports, feature requests, and questions.

    Why Include Them?¶

    • ✅ Get consistent, complete information
    • ✅ Reduce back-and-forth clarification
    • ✅ Help users report issues effectively
    • ✅ Categorize issues automatically

    Location¶

    .github/ISSUE_TEMPLATE/

    Bug Report Template¶

    .github/ISSUE_TEMPLATE/bug_report.md:

    ---
name: Bug Report
about: Report a bug to help us improve
title: '[BUG] '
labels: bug
assignees: ''
---

## Describe the Bug

A clear and concise description of what the bug is.

## To Reproduce

Steps to reproduce the behavior:
1. Go to '...'
2. Click on '....'
3. Scroll down to '....'
4. See error

## Expected Behavior

A clear description of what you expected to happen.

## Screenshots

If applicable, add screenshots to help explain your problem.

## Environment

- OS: [e.g., Ubuntu 22.04]
- Python Version: [e.g., 3.11.0]
- Package Version: [e.g., 1.2.3]

## Additional Context

Add any other context about the problem here.

    Feature Request Template¶

    .github/ISSUE_TEMPLATE/feature_request.md:

    ---
name: Feature Request
about: Suggest an idea for this project
title: '[FEATURE] '
labels: enhancement
assignees: ''
---

## Is your feature request related to a problem?

A clear description of what the problem is. Ex. I'm always frustrated when [...]

## Describe the solution you'd like

A clear description of what you want to happen.

## Describe alternatives you've considered

A clear description of any alternative solutions or features you've considered.

## Additional context

Add any other context or screenshots about the feature request here.

## Would you like to implement this feature?

- [ ] Yes, I'd like to implement this feature
- [ ] No, but I think it's a good idea

    Configuration¶

    .github/ISSUE_TEMPLATE/config.yml:

    blank_issues_enabled: false
contact_links:

    about: Ask questions or discuss ideas
  - name: Documentation
    url: https://your-docs-site.readthedocs.io
    about: Check documentation for answers

    Pull Request Template¶

    Purpose¶

    Guide contributors through PR creation with consistent format.

    Why Include It?¶

    • ✅ Ensure PRs include necessary information
    • ✅ Remind contributors of requirements
    • ✅ Speed up review process
    • ✅ Document changes clearly

    Location¶

    .github/PULL_REQUEST_TEMPLATE.md

    Template¶

    ## Description

Brief description of changes in this PR.

Fixes #(issue)

## Type of Change

- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] Documentation update

## How Has This Been Tested?

Please describe the tests you ran to verify your changes.

- [ ] Test A
- [ ] Test B

## Checklist

- [ ] My code follows the style guidelines of this project
- [ ] I have performed a self-review of my own code
- [ ] I have commented my code, particularly in hard-to-understand areas
- [ ] I have made corresponding changes to the documentation
- [ ] My changes generate no new warnings
- [ ] I have added tests that prove my fix is effective or that my feature works
- [ ] New and existing unit tests pass locally with my changes
- [ ] Any dependent changes have been merged and published

## Screenshots (if applicable)

Add screenshots to demonstrate changes.

## Additional Notes

Any additional information for reviewers.

    SECURITY.md¶

    Purpose¶

    Provide guidelines for reporting security vulnerabilities.

    Why Include It?¶

    • ✅ Encourages responsible disclosure
    • ✅ Provides clear contact information
    • ✅ Shows you take security seriously
    • ✅ Prevents public disclosure of vulnerabilities

    Template¶

    # Security Policy

## Supported Versions

Currently supported versions:

| Version | Supported          |
| ------- | ------------------ |
| 1.1.x   | :white_check_mark: |
| 1.0.x   | :white_check_mark: |
| < 1.0   | :x:                |

## Reporting a Vulnerability

We take security seriously. If you discover a security vulnerability, please follow these steps:

### DO NOT

- ❌ Open a public GitHub issue
- ❌ Discuss the vulnerability publicly
- ❌ Disclose the vulnerability before it's been fixed

### DO

1. **Email** security@THD-Spatial-group.org with:
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact
   - Any suggested fixes

2. **Encrypt** sensitive information using our PGP key (if available)

3. **Wait** for acknowledgment (usually within 48 hours)

### What to Expect

- **Acknowledgment** within 48 hours
- **Assessment** within 1 week
- **Fix timeline** discussed based on severity
- **Credit** in security advisory (if desired)
- **Coordinated disclosure** once fix is available

### Security Update Process

1. Vulnerability confirmed
2. Fix developed and tested
3. Security advisory drafted
4. Patch released
5. Advisory published
6. Users notified

## Security Best Practices for Users

When using this project:

- Keep dependencies up to date
- Use released versions, not development branches
- Follow security guidelines in documentation
- Report suspicious activity

Thank you for helping keep our project secure!

    .gitignore¶

    Purpose¶

    Specify files and directories that Git should ignore.

    Why Include It?¶

    • ✅ Prevent committing sensitive data
    • ✅ Keep repository clean
    • ✅ Avoid committing build artifacts
    • ✅ Exclude environment-specific files

    Python Example¶

    # Byte-compiled / optimized / DLL files

*.py[cod]
*$py.class

# Distribution / packaging

build/
*.egg-info/
.eggs/

# Virtual environments

env/
ENV/

# IDEs

# Testing

    GitHub Actions Workflows¶

    Purpose¶

    Automate testing, building, and deployment.

    Common Workflows¶

    Test Workflow¶

    .github/workflows/tests.yml:

    name: Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.9', '3.10', '3.11', '3.12']

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: ${{ matrix.python-version }}

      - name: Install dependencies
        run: |
          pip install -e .
          pip install pytest pytest-cov

      - name: Run tests
        run: pytest --cov=src tests/

    Documentation Workflow¶

    .github/workflows/docs.yml:

    name: Documentation

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          pip install mkdocs mkdocs-material

      - name: Deploy
        run: mkdocs gh-deploy --force

    cff-version: 1.2.0

  - family-names: "Doe"
    given-names: "Jane"
    orcid: "https://orcid.org/0000-0000-0000-0000"
  - family-names: "Smith"
    given-names: "John"

date-released: 2026-02-19
url: "https://github.com/THD-Spatial/my-project"
