    Best Practices¶

    Follow these best practices to create a high-quality, maintainable, and welcoming open source project.

    Documentation¶

    Write Clear README¶

    Your README is the first thing users see. Make it count.

    Essential Elements¶

    # Project Name

One-sentence description of what your project does.

## Features

- Feature 1
- Feature 2
- Feature 3

## Quick Start

\`\`\`bash
pip install your-project
\`\`\`

\`\`\`python
import your_project
your_project.do_something()
\`\`\`

## Installation

Detailed installation instructions...

## Usage

Comprehensive usage examples...

## Documentation

Link to full documentation...

## Contributing

Link to contributing guidelines...

## License

License information...

    Best Practices¶

    • ✅ Start with a clear, concise description
    • ✅ Include a quick start example that works immediately
    • ✅ Add badges for build status, coverage, version
    • ✅ Use screenshots or GIFs for visual projects
    • ✅ Keep it updated as project evolves
    • ❌ Don't overwhelm with too much detail upfront
    • ❌ Don't assume prior knowledge

    Maintain Comprehensive Documentation¶

    Beyond the README:

    • API Documentation: Document all public functions/classes
    • Tutorials: Step-by-step guides for common tasks
    • Examples: Real-world usage examples
    • FAQ: Common questions and answers
    • Changelog: Track all changes
    • Migration Guides: Help users upgrade between versions

    Use Docstrings¶

    Document your code properly:

    def process_spatial_data(data, resolution=30, method='bilinear'):
    """
    Process spatial data with specified resolution and interpolation method.

    This function resamples spatial data to a target resolution using
    various interpolation methods. It handles both raster and vector data.

    Args:
        data (xarray.DataArray): Input spatial data to process
        resolution (int, optional): Target resolution in meters. Defaults to 30.
        method (str, optional): Interpolation method. Options are 'bilinear',
            'nearest', or 'cubic'. Defaults to 'bilinear'.

    Returns:
        xarray.DataArray: Resampled spatial data at target resolution

    Raises:
        ValueError: If resolution is negative or method is invalid
        TypeError: If data is not an xarray.DataArray

    Examples:
        >>> import xarray as xr
        >>> data = xr.DataArray([...])
        >>> processed = process_spatial_data(data, resolution=10)
        >>> print(processed.shape)
        (100, 100)

    Note:
        This function requires GDAL to be installed for certain interpolation methods.

    See Also:
        - resample_raster: For more advanced resampling options
        - aggregate_data: For aggregation instead of interpolation
    """
    pass

    Code Quality¶

    Follow Style Guidelines¶

    Python¶

    • Follow PEP 8
    • Use Black for formatting
    • Use isort for import sorting
    • Use flake8 or ruff for linting
    # Format code
black .

# Sort imports
isort .

# Check style
flake8 .
# or
ruff check .

    Configuration¶

    pyproject.toml:

    [tool.black]
line-length = 88
target-version = ['py39', 'py310', 'py311']

[tool.isort]
profile = "black"
line_length = 88

[tool.ruff]
line-length = 88
target-version = "py39"

    Write Tests¶

    Good test coverage ensures reliability.

    Test Structure¶

    tests/
├── __init__.py
├── test_core.py
├── test_utils.py
├── test_integration.py
└── conftest.py  # pytest fixtures

    Example Tests¶

    import pytest
from your_project import process_data

def test_process_data_basic():
    """Test basic data processing."""
    input_data = [1, 2, 3, 4, 5]
    result = process_data(input_data)
    assert len(result) == 5
    assert all(isinstance(x, float) for x in result)

def test_process_data_empty():
    """Test empty input handling."""
    result = process_data([])
    assert result == []

def test_process_data_invalid_type():
    """Test invalid input type."""
    with pytest.raises(TypeError):
        process_data("not a list")

@pytest.mark.parametrize("input_val,expected", [
    ([1, 2, 3], 3),
    ([1], 1),
    ([], 0),
])
def test_process_data_length(input_val, expected):
    """Test processing various lengths."""
    result = process_data(input_val)
    assert len(result) == expected

    Run Tests¶

    # Run all tests
pytest

# With coverage
pytest --cov=src --cov-report=html

# Verbose output
pytest -v

# Stop on first failure
pytest -x

# Run specific test
pytest tests/test_core.py::test_function_name

    Version Control¶

    Write Good Commit Messages¶

    Follow conventional commits format:

    <type>(<scope>): <subject>

<body>

<footer>

    Types: - feat: New feature - fix: Bug fix - docs: Documentation only - style: Formatting (no code change) - refactor: Code restructuring - test: Adding/updating tests - chore: Maintenance

    Examples:

    feat(processing): add support for NetCDF4 files

Implements NetCDF4 file reader using xarray. Supports
chunked reading for large files and automatic CRS detection.

Closes #123

    fix(utils): handle missing values in coordinate transformation

Previously, missing values caused errors. Now they are handled
gracefully by filtering before transformation.

Fixes #456

    Branch Strategy¶

    For Small Projects¶

    main (or master)
  └── feature branches

    Create feature branches for each change:

    git checkout -b feature/add-new-feature
git checkout -b fix/bug-description
git checkout -b docs/update-readme

    For Larger Projects¶

    Use Git Flow or GitHub Flow:

    main          # Production-ready code
develop       # Integration branch
  ├── feature/  # New features
  ├── bugfix/   # Bug fixes
  └── hotfix/   # Urgent production fixes

    Keep Commits Atomic¶

    Each commit should be a single logical change:

    ✅ Good: 

    commit: feat(parser): add GeoJSON support
commit: test(parser): add tests for GeoJSON parsing
commit: docs(parser): document GeoJSON usage

    ❌ Bad: 

    commit: add GeoJSON support, fix bug in CSV parser, update README

    Dependency Management¶

    Pin Dependencies¶

    Create appropriate requirements files:

    requirements.txt (exact versions for reproducibility): 

    numpy==1.26.4
pandas==2.2.0
xarray==2024.1.1

    requirements.in (flexible versions for library): 

    numpy>=1.23
pandas>=2.0
xarray>=2023.1

    Use pip-tools: 

    pip-compile requirements.in

    Document System Dependencies¶

    Don't forget non-Python dependencies:

    ## System Requirements

### Ubuntu/Debian
\`\`\`bash
sudo apt-get install libgdal-dev libproj-dev
\`\`\`

### macOS
\`\`\`bash
brew install gdal proj
\`\`\`

### Windows
Download and install:
- GDAL: https://gdal.org/download.html
- PROJ: https://proj.org/download.html

    Use Environment Files¶

    For conda environments:

    name: myproject
channels:
  - conda-forge
dependencies:
  - python=3.11
  - numpy>=1.23
  - pandas>=2.0
  - xarray>=2023.1
  - gdal>=3.6
  - pip
  - pip:
    - some-pip-only-package

    Release Management¶

    Semantic Versioning¶

    Follow SemVer:

    MAJOR.MINOR.PATCH

1.0.0 → 1.0.1 (patch: bug fixes)
1.0.1 → 1.1.0 (minor: new features, backwards compatible)
1.1.0 → 2.0.0 (major: breaking changes)

    Create Releases¶

    # Tag release
git tag -a v1.0.0 -m "Release version 1.0.0"
git push origin v1.0.0

# GitHub release
# Go to Releases → Draft new release → Select tag → Publish

    Release Checklist¶

    • [ ] All tests passing
    • [ ] Documentation updated
    • [ ] CHANGELOG updated
    • [ ] Version numbers updated
    • [ ] Tag created
    • [ ] Release notes written
    • [ ] PyPI package published (if applicable)
    • [ ] GitHub release created
    • [ ] Users notified

    Community Building¶

    Respond Promptly¶

    • Acknowledge issues within 1-2 business days
    • Respond to PRs within 1 week
    • Even "I'll look at this next week" is better than silence

    Be Welcoming¶

    • Thank contributors for their efforts
    • Be patient with beginners
    • Provide constructive feedback
    • Assume good intentions

    Encourage Contributions¶

    • Label good first issues: good first issue
    • Mark help wanted: help wanted
    • Provide clear contribution guidelines
    • Mentor new contributors

    Examples¶

    Thank you for opening this issue! This is a great find. 
We'll investigate and get back to you soon.

---

Thanks for the PR! This looks promising. Before we can merge, 
could you add tests for the new functionality? Let me know 
if you need any help with that.

---

Welcome to your first contribution! 🎉 The changes look good. 
I've left a few minor comments. Don't worry – these are just 
suggestions to help improve the code. Great job!

    Security¶

    Never Commit Secrets¶

    Use environment variables:

    import os

API_KEY = os.getenv('API_KEY')
DATABASE_URL = os.getenv('DATABASE_URL')

    .env file (gitignored): 

    API_KEY=your-secret-key
DATABASE_URL=postgresql://...

    Scan for Secrets¶

    # Install and run git-secrets
brew install git-secrets
git secrets --scan-history

# Or use truffleHog
pip install trufflehog
trufflehog --regex --entropy=True .

    Keep Dependencies Updated¶

    # Check outdated packages
pip list --outdated

# Update with safety check
pip install --upgrade package-name

# Check for security vulnerabilities
pip install safety
safety check

    Performance¶

    Profile Before Optimizing¶

    Don't guess – measure:

    import cProfile
import pstats

# Profile code
profiler = cProfile.Profile()
profiler.enable()

# Your code here
result = expensive_function()

profiler.disable()
stats = pstats.Stats(profiler)
stats.sort_stats('cumtime')
stats.print_stats(10)  # Top 10 functions

    Document Performance Requirements¶

    ## Performance

- Processing 1 GB of data: ~30 seconds on a standard laptop
- Memory usage: ~2x input data size
- Recommended: 16 GB RAM for datasets >4 GB
