    Open Source Readiness Checklist¶

    This comprehensive checklist ensures your repository meets all requirements for open-sourcing under the THD-Spatial group. Work through each section systematically before making your repository public.

    Using This Checklist

    A markdown version of this checklist is available in OPEN_SOURCE_CHECKLIST.md in the repository root. Copy it to your project and check off items as you complete them.

    ‚úÖ Essential Requirements¶

    These items are mandatory for all THD-Spatial public repositories.

    License¶

    • [ ] LICENSE file created in repository root
    • [ ] Appropriate license chosen (see guide)
    • [ ] License text copied into LICENSE file
    • [ ] Year and copyright holder updated
    • [ ] License committed to repository
    • [ ] License mentioned in README

    Critical

    Without a license, you are not allowed to make your repository public under THD-Spatial group.

    Resources: - License Guidelines - choosealicense.com

    README¶

    • [ ] README.md file created in repository root
    • [ ] Project title and description included
    • [ ] Project purpose clearly explained
    • [ ] Key features listed
    • [ ] Installation instructions provided
    • [ ] Usage examples included
    • [ ] Dependencies documented
    • [ ] Contribution guidelines referenced
    • [ ] License mentioned
    • [ ] README formatted with proper markdown
    • [ ] Badges added (optional: build status, coverage, version)

    Example Sections: 

    # Project Name
Description

## Features
## Installation
## Usage
## Contributing
## License

    Contributing Guidelines¶

    • [ ] CONTRIBUTING.md file created in repository root
    • [ ] Issue reporting process documented
    • [ ] Pull request submission process documented
    • [ ] Development setup instructions included
    • [ ] Coding standards and best practices outlined
    • [ ] Testing requirements specified
    • [ ] Guidelines for writing commit messages included
    • [ ] Code review process explained
    • [ ] CONTRIBUTING.md linked in README

    Resources: - Contributing Guidelines Documentation

    Code of Conduct¶

    • [ ] CODE_OF_CONDUCT.md file created in repository root
    • [ ] Contributor Covenant (or equivalent) adopted
    • [ ] Contact information for reporting issues updated
    • [ ] Enforcement process defined
    • [ ] CODE_OF_CONDUCT.md linked in README or documentation

    Resources: - Code of Conduct Guidelines - Contributor Covenant

    üóÇÔ∏è Data Management¶

    Git LFS (Large File Storage)¶

    Required if your repository contains large data files (>10 MB).

    • [ ] Git LFS installed on local machine
    • [ ] Git LFS initialized in repository (git lfs install)
    • [ ] Large file types tracked:
    • [ ] NetCDF files (*.nc)
    • [ ] GeoTIFF files (*.tif, *.tiff)
    • [ ] HDF5 files (*.hdf, *.h5)
    • [ ] Other data formats (specify)
    • [ ] .gitattributes file committed to repository
    • [ ] Large files added and committed properly
    • [ ] LFS usage documented in README
    • [ ] Release strategy for LFS files documented (if applicable)

    Important

    Files tracked by Git LFS are not included in release assets by default. Plan accordingly.

    Resources: - Git LFS Documentation - Git LFS Official Site

    These items significantly improve project quality and maintainability.

    CHANGELOG¶

    • [ ] CHANGELOG.md file created
    • [ ] Version history documented
    • [ ] Changes formatted following Keep a Changelog
    • [ ] Sections include: Added, Changed, Deprecated, Removed, Fixed, Security

    Example Structure: 

    # Changelog

## [Unreleased]

## [1.0.0] - 2026-02-19
### Added
- Initial release

    • [ ] .github/ directory created

    • [ ] Bug report template
    • [ ] Feature request template
    • [ ] Question/Support template (optional)
    • [ ] Pull request template created
    • [ ] Templates include appropriate sections and checklists

    Code Owners¶

    • [ ] CODEOWNERS file created in root or .github/ directory
    • [ ] Code owners assigned for different parts of repository
    • [ ] Teams or individuals properly specified
    • [ ] Path patterns correctly defined

    Example: 

    # Default owners
* @THD-Spatial/maintainers

# Specific paths
/docs/ @THD-Spatial/docs-team
*.py @THD-Spatial/python-developers

    Security Policy¶

    • [ ] SECURITY.md file created
    • [ ] Security vulnerability reporting process documented
    • [ ] Contact information for security issues provided
    • [ ] Supported versions documented
    • [ ] Response timeline expectations set

    Support Guidelines¶

    • [ ] SUPPORT.md file created
    • [ ] Support channels documented:
    • [ ] Issue tracker
    • [ ] Email contact
    • [ ] Discussion forum (if applicable)
    • [ ] Response time expectations set (if applicable)
    • [ ] Common questions addressed

    üîç Quality Checks¶

    Documentation Quality¶

    • [ ] All documentation reviewed for clarity
    • [ ] Spelling and grammar checked
    • [ ] Links tested and working
    • [ ] Code examples tested and verified
    • [ ] Screenshots/diagrams up to date (if applicable)
    • [ ] API documentation complete (if applicable)

    Tools: - Spell checker (VS Code extension, grammarly) - Link checker - Markdown linter

    Repository Settings¶

    • [ ] Repository description added on GitHub
    • [ ] Topics/tags added for discoverability
    • [ ] Website/documentation link added (if applicable)
    • [ ] Default branch configured (typically main or master)
    • [ ] Branch protection rules configured:
    • [ ] Require pull request reviews
    • [ ] Require status checks to pass
    • [ ] Restrict direct pushes to main branch
    • [ ] Issue templates enabled
    • [ ] Discussions enabled (optional)

    Code Quality¶

    • [ ] Code reviewed and cleaned up
    • [ ] Sensitive information removed:
    • [ ] No API keys
    • [ ] No passwords
    • [ ] No tokens or credentials
    • [ ] No personal data
    • [ ] No internal URLs
    • [ ] .gitignore properly configured
    • [ ] Dependencies documented in requirements.txt, environment.yml, etc.
    • [ ] Build/test instructions included
    • [ ] Tests passing (if test suite exists)
    • [ ] Code follows style guidelines
    • [ ] No commented-out code or TODOs in final version

    Scan for Secrets: 

    # Use tools like:
git-secrets, truffleHog, or GitHub's secret scanning

    Testing¶

    • [ ] Repository cloned to fresh location
    • [ ] Installation instructions followed exactly
    • [ ] All dependencies installed successfully
    • [ ] Project builds successfully from scratch
    • [ ] Examples and tutorials run without errors
    • [ ] Tests pass (if applicable)

    Review¶

    • [ ] All essential requirements completed
    • [ ] All recommended items considered
    • [ ] Repository reviewed by at least one other person
    • [ ] All links and references verified
    • [ ] Documentation is comprehensive and clear
    • [ ] Project name and branding consistent throughout

    Communication¶

    • [ ] Team members notified about repository availability
    • [ ] Internal documentation updated (if applicable)
    • [ ] Social media announcement prepared (optional)
    • [ ] Research group informed
